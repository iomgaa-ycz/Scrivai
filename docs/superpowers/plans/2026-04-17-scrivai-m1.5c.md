````markdown
# 变电站技术监督要点(合成示例)

本指南面向 220 kV 及以下变电站,给出 10 类核心技术监督项目的要点描述,供
审核 Agent 对照执行。本文件为合成示例,用于单元/集成测试,非权威规程。

---

## 1. 绝缘电阻测试(insulation_resistance)

对一次设备(变压器、断路器、互感器、电缆)在停电工况下测量绝缘电阻,
使用 2500 V 兆欧表,记录 1 min / 10 min 读数,要求 R60/R15 ≥ 1.3。
测试数据要与前次数据横向对比,下降超过 30% 应深入分析水分或污秽。

## 2. 接地连续性(grounding_continuity)

主接地网、二次设备外壳、避雷器下引线的接地连续性,用低电阻测试仪,
要求接地点至主网的电阻 ≤ 0.2 Ω;末端设备到接地点电阻 ≤ 0.5 Ω。
接地引下线若存在断股、锈蚀、裸铜外露,应列为整改项。

## 3. 变压器油位(transformer_oil_level)

主变压器储油柜油位应在温度-油位曲线的正常带内,环境温度 20 ℃
对应的油位标尺应在 1/2 ~ 2/3 区间。油位异常低提示渗漏,异常高
提示油温过高或呼吸器堵塞,两类均需停机处理。

## 4. 断路器动作试验(circuit_breaker_operation)

每年应做一次分合闸时间测试,分闸时间 ≤ 30 ms,合闸时间 ≤ 80 ms,
三相同期性 ≤ 2 ms。储能电机电流曲线应平滑,无尖峰抖动。试验记录
保存至设备台账并与历史数据对比。

## 5. 互感器校验(ct_pt_calibration)

电流互感器(CT)与电压互感器(PT)每 6 年做一次变比 + 角差 + 比差
校验,CT 精度等级 0.2S 级时变比误差 ≤ ±0.2%,PT 比差 ≤ ±0.3%。
校验前应解开二次侧接线,校验后必须复装并拧紧力矩合格。

## 6. 避雷器检测(surge_arrester_inspection)

金属氧化物避雷器(MOA)每年至少一次带电测试:全电流、阻性电流、
泄漏电流,阻性电流分量 ≤ 全电流的 25%。放电计数器应正常记录,
若 1 年内累计 3 次动作,需做停电性能试验。

## 7. 二次回路接线(secondary_circuit_wiring)

二次回路(保护、测量、控制)的接线柜门应锁闭,标签清晰且与图纸
一致,端子排无虚接、氧化。关键信号(保护 GOOSE/SV)的 IEC 61850
通讯应每月抽检,VLAN 划分与 MAC 过滤表应与设计文件一致。

## 8. 保护定值(protection_settings)

主保护 / 后备保护的定值单应与调度下发版本一致,每次定值修改由
运行人员在「保护定值单管理系统」中双人复核。CPU 板时钟偏差
应 ≤ 5 ms,失同步告警应接入监控后台且响应时间 ≤ 1 min。

## 9. 监控(SCADA)系统(scada_monitoring)

站控层监控主机应双机热备,主/备切换时长 ≤ 3 s,历史数据库每日
归档。遥信、遥测、遥控、遥调四遥信息的覆盖率应 100%,事件
顺序记录(SOE)的时标精度 ≤ 1 ms。

## 10. 运维记录完整性(om_records)

巡视记录、操作票、工作票、缺陷记录、试验报告五类台账需纸质 +
电子双份存档,保存期 ≥ 5 年。当班期间发现的二类及以上缺陷须
24 小时内上报生产管理系统,一类缺陷须立即上报并启动应急预案。
````

```python
"""一次性构造 tests/fixtures/m1_e2e/workpaper_template.docx(docxtpl 3 占位符)。

    / "workpaper_template.docx"

    doc.add_paragraph("{{ audit_summary }}")

    print(f"built: {FIXTURE}")


if __name__ == "__main__":
    build()
```

```python
# ─── 主 E2E 测试 ────────────────────────────────────────────


@pytest.mark.asyncio
@pytest.mark.skipif(
    not os.environ.get("ANTHROPIC_AUTH_TOKEN"),
    reason="real SDK E2E; requires ANTHROPIC_AUTH_TOKEN in .env",
)
async def test_m1_end_to_end_extractor_auditor_generator(tmp_path: Path) -> None:
    """三 PES 串联 + trajectory / archive 验证,产出 markdown 报告。"""

    # ── 公共基础设施 ────────────────────────────────────────
    workspaces_root = tmp_path / "workspaces"
    archives_root = tmp_path / "archives"
    trajectory_db = tmp_path / "trajectories.sqlite"

    ws_manager = build_workspace_manager(
        workspaces_root=workspaces_root,
        archives_root=archives_root,
    )
    store = TrajectoryStore(trajectory_db)

    # 项目 root(提供 skills/agents git metadata 抓取点);用当前 repo 即可
    project_root = Path(__file__).resolve().parents[2]

    model = ModelConfig(
        model=os.getenv("SCRIVAI_DEFAULT_MODEL") or "glm-5.1",
        provider=os.getenv("SCRIVAI_DEFAULT_PROVIDER") or "glm",
    )

    def _hook_manager() -> HookManager:
        hm = HookManager()
        hm.register(TrajectoryRecorderHook(store))
        return hm

    ts = datetime.now().strftime("%Y%m%d_%H%M%S")
    run_ids = {
        "extractor": f"e2e_{ts}_extractor",
        "auditor": f"e2e_{ts}_auditor",
        "generator": f"e2e_{ts}_generator",
    }
    report_lines: list[str] = [
        f"# M1 E2E 集成报告 — {ts}",
        "",
        f"- guide: `{GUIDE_PATH}`",
        f"- checkpoints: `{CHECKPOINTS_PATH}`",
        f"- template: `{TEMPLATE_PATH}`",
        f"- workspaces: `{workspaces_root}`",
        f"- archives: `{archives_root}`",
        f"- trajectory db: `{trajectory_db}`",
        "",
    ]

    # ── Run 1: ExtractorPES ─────────────────────────────────
    ex_ws = ws_manager.create(WorkspaceSpec(
        run_id=run_ids["extractor"],
        project_root=project_root,
    ))
    shutil.copy(GUIDE_PATH, ex_ws.data_dir / "guide.md")

    ex_pes = ExtractorPES(
        config=load_pes_config(Path("scrivai/agents/extractor.yaml")),
        model=model,
        workspace=ex_ws,
        hooks=_hook_manager(),
        trajectory_store=store,
        runtime_context={"output_schema": _ExtractorOutput},
    )
```

````markdown
## Examples

### Example 1 — 审核场景:查围标规则 + 取全文

```bash
scrivai-cli library search --type rules --query "围标串标的认定标准" --top-k 3
# → {"hits": [{"chunk_id": "rule-42::chunk-0", "score": 0.81, "text": "围标的认定…", "metadata": {"title": "招投标法 §32"}}, …]}

qmd document get --collection rules --id rule-42::chunk-0
# → {"id": "rule-42::chunk-0", "markdown": "## 围标的认定\n围标是指…", "metadata": {"title": "招投标法 §32"}, "chunk_count": 1}
```

### Example 2 — 生成场景:带 metadata filter 的搜索

```bash
scrivai-cli library search --type templates --query "验收报告" --filters '{"category": "高压试验"}'
```
Filter 透传到 qmd,只返回 metadata.category == "高压试验" 的 chunk。若不知 metadata key 取值,先跑 `scrivai-cli library list` 抽样。

### Example 3 — 抽取场景:列全部 entry id 预览可用材料

```bash
scrivai-cli library list --type cases
# → {"entry_ids": ["case-2024-001", "case-2024-002", …]}

scrivai-cli library get --type cases --entry-id case-2024-001
# → {"entry_id": "case-2024-001", "markdown": "## 2024 年 XX 变电站审计整改案…", "metadata": {…}}
```

## Errors

所有命令失败时写 JSON 到 stderr + exit 非 0。常见错误与处置:

| 场景 | stderr JSON 片段 | 处置 |
|---|---|---|
| 集合不存在 | `{"error": "collection 'rules' not found"}` | 改 `--type`;不要自行创建 |
| entry id 不存在 | `{"error": "entry 'xxx' not found"}` | 先 `library list` 拿现有 id;或用 search |
| filter JSON 语法错 | `{"error": "invalid --filters: …"}` | 引号用单引号包外层双引号包内层 |
| qmd 未初始化 | `{"error": "qmd not initialized: ~/.scrivai/qmd.db"}` | 报错退出;业务层应先建 db |

**重要**:遇到非零 exit,立即读 stderr 的 JSON,不要忽略继续;你的下一步很可能依赖这个失败的命令结果。
````

````markdown
## Errors

docxtpl **不会**为缺失 key 报错 — 渲染为空字符串。因此框架在 `summarize` 后会:

| 校验失败 | 现象 | 处置 |
|---|---|---|
| `context_schema.model_validate` 失败 | summarize 抛 `response_parse_error` | 补齐 pydantic schema 要求的字段,再写 output.json |
| 占位符覆盖率 < 100% | execute 抛 `output_validation_error` | 漏 placeholder 的 findings 文件必须补上 |
| 循环占位符 content 类型错 | docxtpl 渲染结果空 | 循环占位符的 `content` 必须是 list,不是 str |

**绝不**用 Python 的 `None` 做占位符值 — 会被 docxtpl 渲染成字符串 `"None"`,肉眼不易察觉。用 `""` 或具体 sentinel 字符串。
````
