### 4.1 Workspace (T0.4)

#### 4.1.1 文件布局


#### 4.1.2 类与 API

```python
# scrivai/workspace/manager.py(简化骨架)

import sys
if sys.platform == "win32":
    raise ImportError("scrivai.workspace 仅支持 POSIX(fcntl 不可用)")

import fcntl, json, shutil, subprocess
from datetime import datetime, timezone
from pathlib import Path
from scrivai.exceptions import WorkspaceError
from scrivai.models.workspace import (
    WorkspaceManager, WorkspaceSpec, WorkspaceHandle, WorkspaceSnapshot,
)


class LocalWorkspaceManager:
    """符合 WorkspaceManager Protocol 的本地文件系统实现(POSIX only)。
    通过 build_workspace_manager 工厂构造,不直接对外暴露。"""

    def __init__(self, workspaces_root: Path, archives_root: Path) -> None:
        self.workspaces_root = workspaces_root.expanduser().resolve()
        self.archives_root = archives_root.expanduser().resolve()
        self.workspaces_root.mkdir(parents=True, exist_ok=True)
        self.archives_root.mkdir(parents=True, exist_ok=True)

    def create(self, spec: WorkspaceSpec) -> WorkspaceHandle: ...
    def archive(self, handle: WorkspaceHandle, success: bool) -> Path: ...
    def cleanup_old(self, days: int = 30) -> None: ...

    # 内部辅助
    def _git_hash(self, path: Path) -> str | None: ...    # subprocess git rev-parse;失败 None
    def _git_is_dirty(self, path: Path) -> bool: ...      # git status --porcelain;失败 False
    def _acquire_lock(self, run_id: str) -> int: ...      # fcntl LOCK_EX|LOCK_NB
    def _release_lock(self, lock_fd: int, run_id: str) -> None: ...


def build_workspace_manager(
    workspaces_root: Path | str = "~/.scrivai/workspaces",
    archives_root: Path | str = "~/.scrivai/archives",
) -> WorkspaceManager:
    """构造默认 LocalWorkspaceManager。返回类型用 Protocol。"""
    return LocalWorkspaceManager(Path(workspaces_root), Path(archives_root))
```

#### 4.1.3 关键决策

| 项 | 决策 | 理由 |
|---|---|---|
| 具体类命名 | `LocalWorkspaceManager`(私有) | 暗示"本地文件系统";给未来 `RemoteWorkspaceManager` 留空间 |
| 暴露策略 | 只暴露 factory,不暴露具体类 | 调用方只见 Protocol,符合依赖倒置 |
| `_git_hash` 失败 | 返回 `None` | project_root 可能不是 git 仓库;不阻塞 workspace 创建 |
| `_git_is_dirty` 失败 | 返回 `False` | 安全默认 |
| `create` 流程 | 完全按 design §5.2 伪代码 | 加 lock → exists 检查 → mkdir → copytree(skills/agents) → copy data_inputs → snapshot → meta.json → 释放 lock |
| meta.json 内容 | run_id / project_root / data_inputs / extra_env / snapshot.model_dump() | design §5.2;`meta_path` 不进 WorkspaceHandle 字段(M0 已确认),通过 `handle.root_dir / "meta.json"` 派生 |
| 归档格式 | `.tar.gz`,文件名 `<run_id>.tar.gz` | design §4.13 不变量 2 |
| 归档内容 | `.claude/ + data/ + output/ + logs/ + meta.json` | design §4.13 不变量 2 |
| 失败标记 | workspace root 下写空文件 `.failed` | 不动原目录,`cleanup_old` 通过该标记识别 |
| `cleanup_old(days)` 范围 | 同时清 `archives/<run>.tar.gz` 与 `workspaces/<run>/`(若有 `.failed`) | TD.md T0.4 DoD |
| 锁文件位置 | `workspaces_root / f".{run_id}.lock"` | design §5.2 |
| 锁释放 | `finally` 块 unlock + close + `unlink(missing_ok=True)` | design §5.2 |

#### 4.1.4 契约测试 6 个(`tests/contract/test_workspace.py`)

| 测试 | 验证 |
|---|---|
| `test_create_directory_structure` | `working/.claude/{skills,agents}` + `data/` + `output/` + `logs/` + `meta.json` 存在,`WorkspaceHandle` 字段对齐 |
| `test_snapshot_preserves_skills` | 创建 ws 后修改 `project_root/skills/SKILL.md` 不影响 ws 内容(快照独立性) |
| `test_concurrent_create_rejected` | 多线程 / 多进程同时 `create(run_id="X")` —— 后者 fcntl 拿锁失败,抛 `WorkspaceError("...locked")` |
| `test_force_recreate` | `force=False` + 已存在 → `WorkspaceError`;`force=True` + 已存在 → 删除重建 |
| `test_archive_portability` | `archive(success=True)` 产出 `.tar.gz`,拷到 `/tmp/x/` 解压后 `.claude/skills` + `data/` + `meta.json` 完整;原目录已删 |

#### 4.1.5 错误处理

所有 workspace 操作失败统一抛 `WorkspaceError`(M0 已声明)。具体场景:

| 场景 | message |
|---|---|
| `create` 时 run_id 已存在且 `force=False` | `f"workspace already exists: {root}"` |
| `create` 时 fcntl 锁失败 | `f"workspace {run_id} is locked"` |
| `create` 时 project_root 不存在 | `f"project_root not found: {project_root}"` |
| `archive` 时 handle 对应目录已不在 | `f"workspace not found: {handle.root_dir}"` |

#### 4.2.2 类与 API

```python
# scrivai/pes/hooks.py(简化骨架)

import pluggy
from loguru import logger
from scrivai.models.pes import (
    HookContext, RunHookContext, PhaseHookContext, PromptHookContext,
    PromptTurnHookContext, FailureHookContext, OutputHookContext, CancelHookContext,
)

# 模块级标记 — 与 Herald2 完全对称(只是 namespace 改 scrivai_pes)
hookspec = pluggy.HookspecMarker("scrivai_pes")
hookimpl = pluggy.HookimplMarker("scrivai_pes")


class PESHookSpec:
    """9 个 hook 触点的规范声明(对应 design §4.3 表格)。"""

    @hookspec
    def before_run(self, context: RunHookContext) -> None: ...
    @hookspec
    def before_phase(self, context: PhaseHookContext) -> None: ...
    @hookspec
    def before_prompt(self, context: PromptHookContext) -> None: ...
    @hookspec
    def after_prompt_turn(self, context: PromptTurnHookContext) -> None: ...
    @hookspec
    def after_phase(self, context: PhaseHookContext) -> None: ...
    @hookspec
    def on_phase_failed(self, context: FailureHookContext) -> None: ...
    @hookspec
    def on_output_written(self, context: OutputHookContext) -> None: ...
    @hookspec
    def on_run_cancelled(self, context: CancelHookContext) -> None: ...
    @hookspec
    def after_run(self, context: RunHookContext) -> None: ...


class HookManager:
    """pluggy.PluginManager 轻量封装。"""

    def __init__(self) -> None:
        self._mgr = pluggy.PluginManager("scrivai_pes")
        self._mgr.add_hookspecs(PESHookSpec)

    def register(self, plugin: object, name: str | None = None) -> None:
        self._mgr.register(plugin, name=name)

    def dispatch(self, hook_name: str, context: HookContext) -> None:
        """同步 dispatch;插件异常**冒泡**(由 BasePES 决定怎么处理)。"""
        getattr(self._mgr.hook, hook_name)(context=context)

    def dispatch_non_blocking(self, hook_name: str, context: HookContext) -> None:
        """非阻塞 dispatch;插件异常仅 loguru.exception,不冒泡。"""
        try:
            self.dispatch(hook_name, context)
        except Exception:
            logger.exception("Hook 执行失败 [hook=%s]", hook_name)
```

#### 4.2.3 关键决策

| 项 | 决策 | 理由 |
|---|---|---|
| 同步/非阻塞二分 | **由 BasePES (M0.5) 决定调哪个** | HookManager 只提供两个方法,**不知道**哪些 hook 是哪种;design §4.3 表格的"调用模式"是 BasePES 责任 |
| `register(plugin, name)` | 与 Herald2 完全对称 | 同框架 mental model |
| 异常处理 | 同步透传;非阻塞 catch + `loguru.exception` | design §4.3 异常传播矩阵 |
| `@hookimpl` | 模块级 import:`from scrivai.pes.hooks import hookimpl` | Herald2 / pluggy 标准 |
| 顶层 `__init__.py` 暴露 | `HookManager` + `hookimpl` | TD.md M0.25 DoD |
| HookContext 来源 | 全部 import 自 `scrivai.models.pes`(M0 已定义) | 不重复定义 |

#### 4.2.4 契约测试 5 个(`tests/contract/test_hooks.py`)

| 测试 | 验证 |
|---|---|
| `test_nine_hook_names_registered` | 9 个 hookspec 都能 register/dispatch |
| `test_hook_registration_and_dispatch` | 1 plugin 注册 → dispatch 触发其方法,context 透传无失真 |
| `test_sync_dispatch_propagates_exception` | plugin 抛 ValueError → `dispatch()` 直接抛出 |
| `test_nonblocking_dispatch_catches_exception` | plugin 抛 ValueError → `dispatch_non_blocking()` 不抛、loguru 记一次 |
| `test_hook_ordering` | 多 plugin 注册,assert pluggy 实际调用顺序(LIFO) |

注:`test_hook_ordering` 直接 assert pluggy 默认行为(LIFO,后注册先调)。若 BasePES (M0.5) 需要 FIFO,届时加 `tryfirst=True / trylast=True` 调整,M0.25 不预先优化。

#### 4.3.2 类与 API

```python
# scrivai/trajectory/store.py(简化骨架)

import asyncio, sqlite3, os
from pathlib import Path
from typing import Any
from loguru import logger
from scrivai.exceptions import TrajectoryWriteError
from scrivai.models.trajectory import TrajectoryRecord, FeedbackRecord

DEFAULT_DB_FALLBACK = "~/.scrivai/trajectories.sqlite"


class TrajectoryStore:
    """SQLite 单文件 trajectory 存储。所有 record_* / get_* / list_* 都是 async。
    内部用 asyncio.to_thread 把同步 sqlite3 包装成 async。"""

    def __init__(self, db_path: Path | str | None = None) -> None:
        self.db_path: Path | str
        if db_path == ":memory:":
            self.db_path = ":memory:"
            self._memory_conn: sqlite3.Connection | None = sqlite3.connect(
                ":memory:", check_same_thread=False
            )
        else:
            if db_path is None:
                # 实时读 env(允许测试 monkeypatch.setenv 后构造新实例即生效)
                db_path = os.environ.get("SCRIVAI_TRAJECTORY_DB", DEFAULT_DB_FALLBACK)
            self.db_path = Path(db_path).expanduser()
            self._memory_conn = None
        self._init_schema()

    # ── async record (写)
    async def start_run(self, run_id, pes_name, model_name, provider, sdk_version,
                        skills_git_hash, agents_git_hash, skills_is_dirty,
                        task_prompt, runtime_context) -> None: ...
    async def record_phase_start(self, run_id, phase_name, phase_order, attempt_no) -> int: ...
    async def record_turn(self, phase_id, turn_index, role, content_type, data) -> int: ...
    async def record_tool_call(self, turn_id, tool_name, tool_input, tool_output,
                               status, duration_ms) -> None: ...
    async def record_phase_end(self, phase_id, prompt, response_text, produced_files,
                               usage, error, error_type, is_retryable) -> None: ...
    async def finalize_run(self, run_id, status, final_output,
                           workspace_archive_path, error, error_type) -> None: ...
    async def record_feedback(self, run_id, input_summary, draft_output, final_output,
                              corrections, review_policy_version, source, confidence,
                              submitted_by) -> None: ...

    # ── async get (读)
    async def get_run(self, run_id: str) -> TrajectoryRecord | None: ...
    async def list_runs(self, pes_name: str | None = None,
                        status: str | None = None, limit: int = 50) -> list[TrajectoryRecord]: ...
    async def get_feedback_pairs(self, pes_name: str | None = None,
                                 min_confidence: float | None = None,
                                 limit: int | None = None) -> list[FeedbackRecord]: ...

    # ── 内部
    def _init_schema(self) -> None:
        """同步;__init__ 调一次。CREATE TABLE IF NOT EXISTS + WAL pragmas。"""
    def _get_connection(self) -> sqlite3.Connection:
        """:memory: 返回单例;文件模式 new conn。"""
    async def _execute_with_retry(self, sql: str, params: tuple) -> Any:
        """框架级 1 次 busy 重试(间隔 0.5s);超 → TrajectoryWriteError。"""
```

#### 4.3.3 SQLite Schema(`scrivai/trajectory/schema.py`)

完整对齐 design.md §4.5,包括:

- `runs` 表:`run_id (PK)`, `pes_name`, `model_name`, `provider`, `sdk_version`, `skills_git_hash`, `agents_git_hash`, `skills_is_dirty`, `status (running/completed/failed/cancelled)`, `task_prompt`, `runtime_context (JSON)`, `workspace_archive_path`, `final_output (JSON)`, `error`, `error_type`, `started_at`, `ended_at`
- `phases` 表:`phase_id (PK AUTOINCREMENT)`, `run_id (FK)`, `phase_name`, `phase_order`, `attempt_no`, `prompt`, `response_text`, `produced_files (JSON)`, `usage (JSON)`, `error`, `error_type`, `is_retryable`, `started_at`, `ended_at`, `UNIQUE(run_id, phase_name, attempt_no)`
- `turns` 表:`turn_id (PK)`, `phase_id (FK)`, `turn_index`, `role`, `content_type`, `data (JSON)`, `timestamp`
- `tool_calls` 表:`tool_call_id (PK)`, `turn_id (FK)`, `tool_name`, `tool_input (JSON)`, `tool_output`, `status`, `duration_ms`, `timestamp`
- `feedback` 表:`feedback_id (PK)`, `run_id (FK)`, `input_summary (NOT NULL)`, `draft_output (JSON NOT NULL)`, `final_output (JSON NOT NULL)`, `corrections (JSON)`, `review_policy_version`, `source (default 'human_expert')`, `confidence (default 1.0)`, `submitted_at`, `submitted_by`

索引:`idx_runs_pes_name`, `idx_runs_status`, `idx_runs_started_at`, `idx_phases_run_id`, `idx_turns_phase_id`, `idx_tool_calls_tool_name`, `idx_feedback_run_id`, `idx_feedback_source`。

PRAGMAs(每次 connect 都设):
```sql
PRAGMA journal_mode=WAL;
PRAGMA synchronous=NORMAL;
PRAGMA busy_timeout=3000;    -- 3s,与"框架重试 1 次 0.5s,总预算 ≤ 6.5s"自洽
```

#### 4.3.4 关键决策

| 项 | 决策 | 理由 |
|---|---|---|
| 接口 sync/async | **全 async**(Q5 已定) | BasePES 是 async,免 to_thread 模板 |
| 内部 driver | `sqlite3` + `asyncio.to_thread`(不引入 aiosqlite) | 减少依赖;`to_thread` 性能足够(record P99 < 10ms 设计目标) |
| 连接模型 | **文件模式**:每次 record 新 conn(WAL 保证并发);**`:memory:` 模式**:单 conn 持久 | design 默认 + Fake 必需 |
| `check_same_thread=False`(仅 `:memory:`) | 允许 to_thread worker 用同一 conn | 单 conn 自身在 SQLite 内串行,无竞态 |
| schema init | 首次 `__init__` 同步执行 `CREATE TABLE IF NOT EXISTS` | M0.25 不引入 migration |
| 框架重试 | 仅对 `sqlite3.OperationalError`(busy/locked)重试 1 次(间隔 0.5s);其他错误直抛 | design §4.5 时间预算表 |
| 重试超 → 抛 `TrajectoryWriteError`(M0 已声明) | design §4.5 + 不变量 9 |
| `:memory:` 字符串特判 | 比对字符串 `":memory:"`,绕过 `Path.expanduser()` | 否则 `Path(":memory:")` 被当相对路径 |
| `record_feedback` | 全按 design §4.5 schema;`source` 默认 `"human_expert"`,`confidence` 默认 `1.0` | 与 design 严格一致 |
| env 覆盖 | `SCRIVAI_TRAJECTORY_DB` 在模块加载时读 | `TrajectoryStore()` 不传参时用 env 路径 |

#### 4.3.5 契约测试 8 个(`tests/contract/test_trajectory_store.py`)

| 测试 | 验证 |
|---|---|
| `test_schema_init_creates_five_tables` | 首次 `TrajectoryStore(":memory:")` 后,`sqlite_master` 含 5 张表 |
| `test_full_run_lifecycle_with_attempt_no` | start_run → 3 phase × 2 attempt → record_turn × N → record_tool_call → finalize_run;表行数对齐 |
| `test_phases_unique_constraint` | 同 `(run_id, phase_name, attempt_no)` 重复 → IntegrityError |
| `test_feedback_pair_query_filters_confidence` | confidence={0.5, 0.7, 1.0} 三条 → `get_feedback_pairs(min_confidence=0.7)` 返回 2 条 |
| `test_concurrent_write_safety_two_threads` | 2 协程并发 record_turn 共 100 次 → 100 行落表,无丢失 |
| `test_busy_retry_on_lock` | mock conn 第 1 次抛 `OperationalError("database is locked")`,第 2 次成功 → 整体成功 |
| `test_busy_exhausts_raises_trajectory_write_error` | mock conn 持续抛 busy → 抛 `TrajectoryWriteError` |
| `test_default_path_uses_env_override` | env `SCRIVAI_TRAJECTORY_DB=/tmp/x.db` → `TrajectoryStore()` 用此路径 |

#### 4.3.6 `:memory:` 单连接的并发正确性

`:memory:` 模式:单 conn + `check_same_thread=False`,所有 to_thread worker 共享。SQLite C 层对单 conn 自带串行(每次 query 内部锁)。多 async task 通过 to_thread 落到同一 conn → SQLite 自己排队 → **无竞态、无锁失败**。这是 `test_concurrent_write_safety_two_threads` 必须能过的根本原因。

## 7. 风险与缓解

| 风险 | 等级 | 缓解 |
|---|---|---|
| pluggy 调用顺序 LIFO 与 design 直觉(FIFO)不符 | 中 | `test_hook_ordering` 直接 assert 实际行为;若 BasePES 需要 FIFO,M0.5 加 `tryfirst=True` 调整 |
| `:memory:` 单 conn `check_same_thread=False` 在 to_thread 下产生竞态 | 中 | 实测靠 SQLite C 层串行;`test_concurrent_write_safety_two_threads` 兜底 |
| fcntl 锁文件 `.{run_id}.lock` 在测试间残留 | 低 | finally 块 `unlink(missing_ok=True)`;tmp_path 自动清 |
| WAL 衍生文件(`*.db-wal / *.db-shm`)在 cleanup 时残留 | 低 | 文件模式测试都用 tmp_path;`:memory:` 无此问题 |
| pluggy 异常包装("HookCallError")污染原始 traceback | 低 | `dispatch_non_blocking` 用 `logger.exception` 仍能拿到完整链;同步 dispatch 透传时 BasePES (M0.5) 有责任 unwrap |
| design §4.5 与 TD.md T0.7 在 `busy_timeout` 上自相矛盾(3000 vs 10000) | 低 | **采纳 design**(3000),与"框架重试 1 次 0.5s"配套总预算 ≤ 6.5s 一致;TD.md 后续修订一并校对 |
| `_git_hash` 在非 git 仓库的 project_root 抛而非返 None | 低 | `subprocess.run(check=False)`,returncode != 0 → None |
| Windows 下 fcntl import 失败导致整模块加载崩 | 低 | `manager.py` 顶部 `if sys.platform == "win32": raise ImportError(...)`;契约测试 `pytest.skip` |
