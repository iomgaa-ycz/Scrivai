## 3. Knowledge Libraries（T0.11）

### 3.1 文件结构

```
scrivai/knowledge/
├── __init__.py      # 导出 RuleLibrary, CaseLibrary, TemplateLibrary,
│                    #       build_libraries, build_qmd_client_from_config
├── base.py          # _BaseLibrary(qmd_client, collection_name)
├── rules.py         # RuleLibrary(_BaseLibrary), collection="rules"
├── cases.py         # CaseLibrary(_BaseLibrary), collection="cases"
├── templates.py     # TemplateLibrary(_BaseLibrary), collection="templates"
└── factory.py       # build_qmd_client_from_config + build_libraries
```

### 3.2 `_BaseLibrary` 公共接口

实现 `Library` Protocol（定义在 `scrivai/models/knowledge.py`）。

| 方法 | 签名 | 说明 |
|------|------|------|
| `add` | `(entry_id: str, markdown: str, metadata: dict) -> LibraryEntry` | 写入 qmd chunk；metadata 中注入 `_entry_id` + `_created_at`；重复 entry_id 抛 `ValueError` |
| `get` | `(entry_id: str) -> LibraryEntry \| None` | 按 `metadata._entry_id` 过滤查 qmd |
| `list` | `() -> list[str]` | 返回 collection 内所有 entry_id |
| `delete` | `(entry_id: str) -> None` | 按 metadata 过滤删除 |
| `search` | `(query: str, top_k: int = 5, filters: dict \| None = None) -> list[SearchResult]` | 透传 `qmd.Collection.search()` |

### 3.3 关键设计决策

- entry 元数据**完全持久化在 qmd chunk metadata**，Library 无内存状态（design §4.7）
- `_entry_id` 用 metadata 字段存储（qmd chunk 的 id 是内部 rowid，不做业务 id）
- 三个子类只有 `__init__` 中 `super().__init__(client, "rules"/"cases"/"templates")`，无额外逻辑

### 3.4 工厂函数

```python
def build_qmd_client_from_config(db_path: str | Path) -> QmdClient:
    """封装 qmd.connect，统一路径处理。"""
    return qmd.connect(str(Path(db_path).expanduser()))

def build_libraries(qmd_client: QmdClient) -> tuple[RuleLibrary, CaseLibrary, TemplateLibrary]:
    """一次性构建三兄弟。"""
    return RuleLibrary(qmd_client), CaseLibrary(qmd_client), TemplateLibrary(qmd_client)
```

### 3.5 契约测试

- `test_library_crud`：add → get → list → delete → get returns None

- `test_cross_collection_isolation`：rules 和 cases 各 add 同 entry_id → 互不影响

## 4. IO 工具（T0.12）

### 4.1 文件结构

```
scrivai/io/
├── __init__.py      # 导出 docx_to_markdown, doc_to_markdown,
│                    #       pdf_to_markdown, DocxRenderer
├── convert.py       # 三个转换函数
└── render.py        # DocxRenderer
```

### 4.2 转换函数

| 函数 | 实现 | 外部依赖 |
|------|------|---------|
| `docx_to_markdown(path: str \| Path) -> str` | `subprocess.run(["pandoc", path, "-t", "markdown"])` 捕获 stdout | pandoc 二进制 |
| `doc_to_markdown(path: str \| Path) -> str` | LibreOffice headless 先转 docx，再调 `docx_to_markdown` | libreoffice + pandoc |
| `pdf_to_markdown(path: str \| Path, *, base_url: str = "http://100.81.95.44:7861", timeout: int = 120) -> str` | MonkeyOCR HTTP：POST `/parse` 上传 PDF → 取 `download_url` → 下载 ZIP → 提取 `.md` 文件 | MonkeyOCR Docker 服务 |

**`pdf_to_markdown` 要点**：

- `base_url` 默认硬编码 `"http://100.81.95.44:7861"`，非默认值通过调用方显式传参（业务层可从 YAML 读取）
- 依赖 `requests` 库（加入 `pyproject.toml` dependencies）
- 参考 `Reference/smart-construction-ai/crawler.py` 的 `MonkeyOCRClient`，但不封装为类——MVP 一个函数够用
- 服务不可达 / 响应非 200 / ZIP 中无 `.md` → 抛 `IOError`

### 4.3 DocxRenderer

```python
class DocxRenderer:
    def __init__(self, template_path: str | Path):
        """加载 docxtpl 模板。模板不存在抛 FileNotFoundError。"""

    def list_placeholders(self) -> list[str]:
        """正则提取 {{ xxx }} 占位符名称列表（去重、排序）。"""

    def render(self, context: dict[str, Any], output_path: str | Path) -> Path:
        """渲染模板并写入 output_path。成功返回 Path；失败不留半成品。"""
```

- 基于 `docxtpl`（加入 `pyproject.toml` dependencies）
- `render` 失败时删除可能的半成品文件（`try/finally`）

### 4.4 pyproject.toml 变更

dependencies 新增 `requests>=2.28` 和 `docxtpl>=0.16`。

### 4.5 契约测试

`tests/contract/test_io_smoke.py`（smoke 级别）：

- `test_docx_to_markdown`：用 `python-docx` 创建简单 .docx fixture → 转换 → 返回非空含预期文本
- `test_doc_to_markdown`：`skipif(not shutil.which("libreoffice"))`

### 4.6 IO 与 Agent 的关系

IO 工具是**业务层预处理用**，不是 Agent 运行时工具。默认模式：

```
业务层（Python）                    Agent（workspace 内）
PDF/DOCX/DOC                        
  → 业务层调 pdf_to_markdown()     → Agent 只读 data/tender.md
  → 产出 markdown                  → 专注推理、检索、生成
  → 传入 data_inputs               
```

`available-tools/SKILL.md` **不列** `io` 命令。CLI `io` 子命令存在但面向业务层/脚本使用。

## 5. scrivai-cli（T0.13）

### 5.1 文件结构

```
scrivai/cli/
├── __init__.py        # 空
├── __main__.py        # argparse 顶层路由 → 4 个 group
├── library.py         # library {search, get, list}
├── io_cmd.py          # io {docx2md, doc2md, pdf2md, render}
├── workspace_cmd.py   # workspace {create, archive, cleanup}
└── trajectory_cmd.py  # trajectory {record-feedback, list, get-run, build-eval-dataset}
```

### 5.2 通用约定

- 成功：JSON stdout（`json.dumps(..., ensure_ascii=False, indent=2)`）
- 失败：JSON stderr `{"error": "..."}` + exit 1
- env 回退：`SCRIVAI_PROJECT_ROOT` / `QMD_DB_PATH` / `SCRIVAI_WORKSPACE_ROOT` / `SCRIVAI_ARCHIVES_ROOT` / `SCRIVAI_TRAJECTORY_DB`
- CLI 输出 JSON shape 与对应 Python API `.model_dump(mode="json")` 严格一致

### 5.3 子命令清单

| Group | 子命令 | 核心逻辑 |
|-------|--------|---------|
| `library` | `search --type {rules,cases,templates} --query Q [--top-k 5] [--filters '{}']` | `build_qmd_client_from_config` → `build_libraries` → 调对应 library → JSON |
| | `get --type T --entry-id ID` | |
| | `list --type T [--filters '{}']` | |
| `io` | `docx2md --input P [--output P]` | 调 `docx_to_markdown` → stdout 或写文件 |
| | `doc2md --input P [--output P]` | 调 `doc_to_markdown` |
| | `pdf2md --input P [--output P] [--base-url URL]` | 调 `pdf_to_markdown` |
| | `render --template P --context-json P --output P` | 调 `DocxRenderer` |
| `workspace` | `create --run-id ID --project-root P [--data name=path...] [--env K=V...]` | 调 `build_workspace_manager` → `create` |
| | `archive --run-id ID {--success\|--failed}` | 调 `archive` |
| | `cleanup [--days 30]` | 调 `cleanup_old` |
| `trajectory` | `record-feedback --run-id ID --draft P --final P [--corrections P] [--input-summary S] [--source S] [--confidence F]` | 调 `store.record_feedback` |
| | `list [--pes-name N] [--limit 50]` | 调 `store.list_runs` |
| | `get-run --run-id ID` | 调 `store.get_run` |
| | `build-eval-dataset --pes-name N --output P` | M0.75 占位：`NotImplementedError("M2 实现")` |

## 6. Skills + Agent Profiles（T0.14 + T0.15）

### 6.1 Skills 文件结构

```
skills/
├── available-tools/SKILL.md       ← Agent 可见命令 manifest
├── search-knowledge/SKILL.md      ← library search 使用时机
├── inspect-document/SKILL.md      ← chunk_id 反查原文
└── render-output/SKILL.md         ← docxtpl 上下文约定
```

### 6.2 `available-tools/SKILL.md` 内容范围

只列以下 Agent 可见子命令（**严禁** `workspace` / `trajectory` / `io`）：

```bash
# library — Agent 在 execute 阶段查知识库
scrivai-cli library search --type rules|cases|templates --query <q> [--top-k 5]
scrivai-cli library get    --type rules|cases|templates --entry-id <id>
scrivai-cli library list   --type rules|cases|templates

# qmd — 直接语义检索
qmd search --collection <name> --query <q> [--top-k 5] [--rerank]
qmd document {get,add,delete,list}
qmd collection {info,list}
```

每个命令含参数说明、输出 JSON shape、典型错误。

### 6.3 其他 3 个 Skill

| Skill | 核心内容 |
|-------|---------|
| `search-knowledge` | 使用时机（execute 阶段查法规/案例/模板）+ 调用方式 + 多表述提高召回 + 引用必带 chunk_id |
| `inspect-document` | 从 chunk_id 反查原文段落：`qmd document get --id <chunk_id>` |
| `render-output` | docxtpl 上下文 schema 约定，占位符命名规范 |

所有 Skill 严格按 Anthropic SKILL.md 格式（YAML frontmatter `name` + `description` + Markdown body）。

**Skill 创作标准**：实现时须参考 `Herald2/.claude/skills/skill-creator/SKILL.md` 的 Skill Writing Guide，特别是：

- **Progressive Disclosure**：metadata（~100 words）→ body（<500 lines）→ bundled resources（按需加载）
- **Description 要 "pushy"**：包含 skill 做什么 + 具体触发场景，适度宽泛以避免 undertrigger
- **Writing Style**：explain the why，避免 rigid ALWAYS/NEVER；lean instructions，删除不拉权重的内容
- **Writing Patterns**：用 imperative form，提供具体 examples（input → output 格式）

### 6.4 Agent Profiles

```
agents/
├── extractor.yaml     ← ExtractorPES 默认 PESConfig
├── auditor.yaml       ← AuditorPES 默认 PESConfig
└── generator.yaml     ← GeneratorPES 默认 PESConfig
```

每份含完整 `PESConfig` 结构：`name` / `display_name` / `prompt_text` / `default_skills` / `phases`（plan / execute / summarize 各含 `additional_system_prompt` / `allowed_tools` / `max_turns` / `required_outputs`）。

三份 YAML 都能通过 `load_pes_config(yaml_path)` 校验。
