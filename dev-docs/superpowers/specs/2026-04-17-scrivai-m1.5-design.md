## 4. API 契约

### 4.1 三个 PES 的构造签名(真·Herald2 模式)

```python
class ExtractorPES(BasePES):
    """从文档抽取结构化条目。

    runtime_context 业务字段:
    - output_schema: type[BaseModel]  (必需,summarize 校验用)
    """
    # 不 override __init__,直接继承 BasePES.__init__

class AuditorPES(BasePES):
    """对照 data/checkpoints.json 审核。

    runtime_context 业务字段:
    - output_schema: type[BaseModel]            (必需)
    - verdict_levels: list[str]                 (可选,默认 ["合格","不合格","不适用","需要澄清"])
    - evidence_required: bool                   (可选,默认 True)
    """

class GeneratorPES(BasePES):
    """按 docxtpl 模板生成。

    runtime_context 业务字段:
    - template_path: Path                        (必需)
    - context_schema: type[BaseModel]            (必需)
    - auto_render: bool                          (可选,默认 False)
    """
```

### 4.2 扩展点覆盖矩阵

| 扩展点 | ExtractorPES | AuditorPES | GeneratorPES |
|---|---|---|---|
| `build_execution_context` | 默认 `{}` | 默认 | **override** `plan` 阶段:docxtpl 解析 `template_path` 占位符注入 `context["placeholders"]` |
| `build_phase_prompt` | 默认 | 默认 | 默认 |
| `postprocess_phase_result` | **override** `summarize`:读 output.json → `output_schema.model_validate()` | **override** `summarize`:同 + 每 finding.verdict ∈ verdict_levels + evidence 非空 | **override** `summarize`:同 + `auto_render=True` 时 `DocxRenderer.render` 产 final.docx |
| `validate_phase_outputs` | **override** `execute`:plan.items 与 findings/*.json 对齐 | **override** `execute`:checkpoints.json 的 cp_id 与 findings/*.json 对齐 | **override** `plan`:plan.json 覆盖所有占位符;**override** `execute`:占位符覆盖率 |

未 override 的扩展点沿用 BasePES 默认(`required_outputs` 基础校验仍生效)。

### 4.3 使用示例

```python
from pathlib import Path
from pydantic import BaseModel
from scrivai import (
    AuditorPES, ExtractorPES, GeneratorPES,
    ModelConfig, load_pes_config, build_workspace_manager,
)
from scrivai.models.workspace import WorkspaceSpec

class ExtractItem(BaseModel):
    id: str
    content: str

class ExtractOutput(BaseModel):
    items: list[ExtractItem]

ws = build_workspace_manager().create(WorkspaceSpec(
    run_id="demo-001", project_root=Path("./myproject"),
))
pes = ExtractorPES(
    config=load_pes_config(Path("scrivai/agents/extractor.yaml")),
    model=ModelConfig(model="glm-5.1", provider="glm",
                      base_url="...", api_key="..."),
    workspace=ws,
    runtime_context={"output_schema": ExtractOutput},
)
run = await pes.run(task_prompt="从 data/guide.md 抽取 10 条...")
assert run.status == "completed"
assert isinstance(run.final_output, ExtractOutput)
```

## 5. 数据流与文件契约

### 5.1 Workspace 文件契约(三个 PES 共享)

```
workspaces/<run_id>/
├── working/                           PES Agent 的 cwd
│   ├── .claude/skills/                ← M0.25 快照(SDK 自动读)
│   ├── plan.md / plan.json            plan 阶段产物
│   ├── findings/*.json                execute 阶段每项一文件
│   └── output.json                    summarize 唯一最终输出
├── data/                              业务层预置只读输入
│   └── checkpoints.json               (AuditorPES 专属,业务层写入)
├── output/                            PES 发布产物
│   └── final.docx                     (GeneratorPES auto_render=True 时)
├── logs/
└── meta.json
```

### 5.2 三个 PES 的阶段数据流

**ExtractorPES**:
```
plan     → plan.json: {items_to_extract: [{id, ...}, ...]}
execute  → findings/<id>.json  (每 plan item 一个)
           [validate] plan item.id 集合 ⊆ findings/*.json stem
summarize→ output.json
           [postprocess] output_schema.model_validate(output.json)
```

**AuditorPES**:
```
业务层前置: 写入 workspace/data/checkpoints.json
plan     → plan.json: 逐项审核计划(Agent 读 data/checkpoints.json)
execute  → findings/<cp_id>.json
           [validate] checkpoints.json 的 id 集合 = findings/*.json stem
summarize→ output.json
           [postprocess] output_schema.model_validate +
                        每 finding.verdict ∈ verdict_levels +
                        evidence_required ⇒ evidence 非空
```

**GeneratorPES**:
```
[build_execution_context phase="plan"]
           docxtpl parse(template_path) → context["placeholders"]=[...]
plan     → plan.json: 每占位符的素材计划
           [validate] plan.json 覆盖所有 placeholders
execute  → findings/<placeholder>.json
           [validate] placeholders 集合 = findings/*.json stem
summarize→ output.json: {context: {...}, sections: [...]}
           [postprocess] context_schema.model_validate(output.json.context) +
                        if auto_render: DocxRenderer.render → output/final.docx
```

## 6. 错误处理

### 6.1 故障点 → error_type 映射(零新增,复用 M0.5)

| 故障点 | error_type | is_retryable |
|---|---|---|
| `runtime_context` 必需字段缺失 | `response_parse_error` | False |
| `output.json` 不存在 / JSON 解析失败 | `response_parse_error` | False |
| `output_schema.model_validate` 失败 | `response_parse_error` | False |
| Auditor verdict 不在 verdict_levels | `response_parse_error` | False |
| Auditor evidence_required=True 但 evidence 为空 | `response_parse_error` | False |
| Generator docxtpl 渲染失败 | `response_parse_error` | False |
| Extractor/Auditor/Generator 覆盖率校验失败 | `output_validation_error` | True |
| SDK 层 max_turns / 其他 | `max_turns_exceeded` / `sdk_other` | True |

### 6.2 设计原则

- **不在 `__init__` 早校验 runtime_context**(Herald2 模式):业务字段延迟到 summarize 取值,缺失时抛 PhaseError
- **is_retryable 语义守恒**:pydantic 校验失败不可重试(LLM 再跑也不会"突然对");覆盖率失败可重试(多跑一次可能产出缺的 findings)
